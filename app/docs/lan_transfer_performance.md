# LAN 文件传输性能调查报告

## 结论

WiFi LAN 环境下，将单个大文件拆分为多段并行传输（多 Isolate / 多 TCP 连接）**无法提升传输速度**。
瓶颈在 WiFi 物理带宽，而非 CPU 或软件层面。单个 TCP 连接已经可以跑满 WiFi 链路。

## 测试数据

| 方案 | 文件大小 | 段数 | 各段速度 | 总速度 | 总耗时 |
|------|---------|------|---------|--------|--------|
| 4 段并行 | 190 MB | 4 | 5.6–10.5 MB/s | 20.7 MB/s | 9.2s |
| 单连接（预期） | 190 MB | 1 | ~20 MB/s | ~20 MB/s | ~9.5s |

4 条并行 TCP 流共享同一条 WiFi 无线电链路，每条流分到约 5–10 MB/s，总吞吐仍然是 ~20 MB/s，
与单连接传输几乎相同。

## 原因分析

1. **WiFi 是半双工共享介质**：所有 TCP 连接共用同一个无线电收发器，总带宽固定。
2. **单连接已能饱和**：20 MB/s ≈ 160 Mbps，已接近 2.4GHz WiFi 的实际吞吐上限。
3. **多连接额外开销**：分段需要接收端合并文件（磁盘 I/O），增加了延迟和复杂度。

## 多线程并发的适用场景

| 场景 | 原因 |
|------|------|
| S3 / 公网传输 | 高延迟网络，单连接受 TCP 窗口/拥塞控制限制，多连接能叠加吞吐 |
| 同时传输多个文件 | 减少每个文件的连接建立/等待开销，整体吞吐更高 |
| 企业网络单连接限速 | 部分网络对单连接有带宽限制策略，多连接可绕过 |
| 万兆以太网 (10GbE+) | 单 TCP 流可能无法跑满链路，多流能叠加 |

## 决策

- **LAN 直传**：不再拆分单文件为多段，每个文件使用单个 HTTP 连接传输。
- **多文件并发**：当用户选择多个文件时，最多同时传输 N 个文件（利用接收端 Worker Isolate 池处理并发请求）。
- **接收端 Worker 池保留**：4 个 Worker Isolate 共享端口，处理来自不同发送端的并发连接。

## WiFi 频段对速度的影响

| 频段 | 标准 | 典型实际吞吐 |
|------|------|-------------|
| 2.4 GHz | WiFi 4 (802.11n) | 15–25 MB/s |
| 5 GHz | WiFi 5 (802.11ac) | 40–80 MB/s |
| 5/6 GHz | WiFi 6 (802.11ax) | 60–120 MB/s |

确保两台设备都连接 5GHz 频段是最有效的提速手段。
