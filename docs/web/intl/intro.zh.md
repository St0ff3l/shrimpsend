# ShrimpSend 主要功能介绍（国际）

**适用区域**：国际服务集群（例如 `api.shrimpsend.com` / `ws.shrimpsend.com`）

ShrimpSend 是一个面向个人多设备的消息与文件发送工具。它不是网盘，也不是传统的「上传文件、创建链接、再把链接发给自己」的流程。它更像一个属于你自己设备之间的会话窗口：选择目标设备，然后把文本、剪贴板、图片、视频和大文件直接发送过去。

## 为什么需要 ShrimpSend

现代工作流里，设备越来越多：手机、笔记本、台式机、平板、公司电脑、临时浏览器。真正高频的需求往往很小，但会反复发生：

- 把手机截图发到电脑。
- 把电脑上的安装包发到另一台设备。
- 把浏览器里复制的文字发到手机。
- 不方便安装客户端时（公司电脑、访客设备），打开浏览器就能传；同一局域网里，网页对网页也可以直连互传。
- 把一个大文件发到同一 Wi-Fi 下的电脑。
- 传输中断后继续传，而不是重新开始。
- 安卓和 Windows 同网时，Windows 防火墙常拦截入站连接，容易出现「手机能传给电脑、电脑却发不回手机」的单向连通。
- 家庭网络并不总是同一网段：一台插网线、一台走 Wi-Fi 或二级路由，设备可能落在上下游不同 NAT 层级，看似同网、直连却时通时不通。
- 局域网不可用时，仍然需要一个可靠的兜底路径。

ShrimpSend 的设计目标就是减少这些场景里的摩擦。

## 核心交互：设备会话

ShrimpSend 以设备为中心。你不是先创建一个公开链接，而是把内容发送到某台设备的会话里。这样更适合连续发送：一段文字、几张图、一个压缩包、一个视频，都可以在同一条设备会话中完成。

## 主要功能

### 文本和剪贴板发送

不是所有需求都值得打开网盘或聊天软件。配置片段、链接、命令、临时备注、验证码说明和短文本，都可以直接放进设备会话。

### 文件和大文件发送

ShrimpSend 支持图片、视频、压缩包、安装包和项目文件。大文件在同一局域网内更适合走直连路径；如果传输中断，原生客户端可以从中断位置继续，而不是从头重传。

### 局域网优先

当设备在同一网络中时，ShrimpSend 会优先尝试 LAN / WebRTC 等直接路径。对高频发送和大文件来说，这通常比公网中转更自然。

### S3 兜底

S3 的定位是兜底，而不是替代局域网。当直连不可用、设备跨网络或网络环境受限时，托管对象存储或自建 S3 可以保证文件仍然送达。

### Web 端加入

Web 端适合临时使用：在没有安装客户端的电脑上打开浏览器，也能加入设备会话，和原生客户端之间收发内容。

## 产品对比

不同传输工具的定位不同。ShrimpSend 更关注「自己的多台设备之间，频繁发送文本和文件」这个场景，而不是一次性公开分享或单纯的局域网传输。

| 产品 | 优点 | 局限 | 更适合的场景 |
| --- | --- | --- | --- |
| ShrimpSend | 设备会话模型，适合连续发送文本、剪贴板和文件；同网段优先 LAN / WebRTC；原生客户端支持断点续传；直连不可用时可用托管或自建 S3 兜底；Web 端可临时参与。 | 需要安装客户端才能获得完整体验；Web 端受浏览器能力限制；部分增强能力需要登录。 | 自己的手机、电脑和浏览器之间高频发送；同网络大文件；跨网络时需要可靠兜底。 |
| WeTransfer | 不需要双方安装客户端；适合一次性把文件发给别人；收件人通过链接下载，协作门槛低。 | 更偏「上传到云端生成链接」；不适合自己的多设备连续发送；大文件依赖公网和平台存储；局域网直连不是核心能力。 | 给外部客户、同事或临时收件人分享文件。 |
| LocalSend | 开源、局域网优先、无需账号；同一网络内传输直接、简单。 | 主要依赖局域网可达；跨网络、复杂 NAT、Web 端参与和兜底能力不是重点；设备会话和连续文本/文件发送体验相对弱。 | 家庭、办公室同一 Wi-Fi 下，设备都安装客户端并且网络互通。 |
| 云盘 / Cloud Drive | 文件管理、长期保存、多人共享能力强；适合资料归档。 | 上传、整理、分享链路更重；临时发送文本和零散文件不够轻；局域网直传和发送到设备不是核心体验。 | 长期存储、团队资料管理、跨人共享文件夹。 |
| 聊天软件文件助手 | 随手可用，适合小文本和小文件；用户习惯成本低。 | 单端通常不支持多设备同时登录（例如安卓同一时间只允许一台设备在线），跨设备收发不稳定；文件大小、格式、压缩策略和有效期可能受限制；大文件、断点续传、局域网直传和多设备管理通常不是重点。 | 轻量文本、小图片、小文件的临时转发。 |

## 如何选择

- 如果你要把文件发给外部人员，WeTransfer 或云盘分享更合适。
- 如果所有设备都在同一局域网、只需要偶尔传文件，LocalSend 很直接。
- 如果你每天都在自己的多台设备之间发送文本、截图、安装包、视频和大文件，并且希望局域网优先、失败时有兜底路径，ShrimpSend 更适合。
- 如果你在意文件长期整理和归档，云盘仍然是更合适的长期存储工具；ShrimpSend 更像一个高频发送通道。

## 登录与免登录

原生客户端之间的同局域网传输可以免登录。登录主要用于让 Web 端加入设备会话、同步设备列表和会员状态，并在防火墙、NAT 等单向网络环境下提供服务端辅助发现。

## 文本消息云端存储与加密

当你通过服务端路径同步文本消息（例如已登录、跨网络在多设备间拉取历史）时，我们对云端保存的文本内容采取以下策略：

- **只加密文本正文**：仅对文本类消息的正文内容做服务端加密存储；消息类型、时间、设备信息、会话标识等元数据保持可读，以便正常同步与运维。
- **文件消息保持可读**：文件类消息的相关信息（如文件名、存储位置、大小等）不以相同方式加密，便于后续按对象生命周期做清理与管理。
- **本地不加密**：客户端本地缓存的文本消息保持明文，便于离线查看与本地搜索。
- **不在云端搜索**：我们不在服务端对消息正文做关键词搜索；搜索仅在你设备本地已缓存的消息中进行。

纯局域网直传的文本不会写入云端数据库，因此不受上述云端存储加密策略约束。

## 国际服务特点

国际服务集群面向海外用户，法律主体、支付渠道、会员订阅和托管存储规则按国际版本展示。会员通常使用美元订阅，支付渠道可能包括 Apple、Google Play、Stripe 和 RevenueCat。
